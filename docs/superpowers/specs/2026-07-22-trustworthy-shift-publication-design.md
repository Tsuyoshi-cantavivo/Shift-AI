# 信頼できるシフト公開・変更フロー — 設計

**日付:** 2026-07-22
**対象:** ShiftAI 第1段階（公平性・透明性）

## 1. 目的

ShiftAI を「AIが割り当てる道具」から、スタッフが希望と確定結果の関係を理解し、公開後の変更も追跡できる運用基盤へ進化させる。

この段階では次を必ず守る。

1. AIドラフト、スタッフ希望、公開済みシフトを混同しない。
2. 公開済みのシフトを、本人への説明・通知・記録なしに変更しない。
3. 公開前に安全・労務・公平性の問題を人が確認できる。
4. スタッフは自分の希望、確定結果、差分理由、相談先を確認できる。

## 2. 成功条件

- AI生成とプレビューだけでは `shifts` の確定行を変更しない。
- 公開操作は、対象のドラフト版だけを確定対象にする。期間中の全 `requested` を一括確定しない。
- 休希望は希望データとして保存され、生成・公開の両方で勤務不可として扱われる。
- 公開前に未反映希望、希望外配置、0時間者、学生監督違反、必要人数超過、既存確定シフトへの差分を表示する。
- 公開後の変更は監査ログ、対象者通知、スタッフ側の受領状態を持つ。
- 書込み経路を問わず同じ制約検証を通す。

## 3. 範囲

### 含める

- ドラフトと公開済みシフトの状態分離
- 公開前レビューと個人別の説明
- 公開・変更の監査ログとアプリ内通知
- 自動短縮を「提案」に変更
- 休希望の正しい送受信
- P0の認可・初期管理者対策

### 含めない

- 打刻による実績勤怠、給与確定、外部給与連携
- SMS、メール、LINE等の外部通知到達保証
- 自動的なスタッフ同意の取得。第1段階ではアプリ内受領確認までとする。
- 過去全シフトの完全な履歴復元

## 4. 方針と代替案

### 採用: ドラフトを公開シフトから別テーブルへ分離する

AI生成物を `shifts` の `requested` に混在させず、公開候補を独立した版として保存する。公開時だけ検証済みの差分を `shifts` に反映する。

利点は、希望がドラフト確定に巻き込まれず、公開前後を比較できること。既存の `status` 制約を大きく変更せずに移行できる。

### 不採用: `shifts.status` に `draft` を足す

実装量は少ないが、スタッフ希望も `requested` にあり、理由文字列で判別する現在の曖昧さを残す。公開対象の版、再生成、差分確認を安全に扱えない。

### 不採用: 勤怠・給与まで同時に全面再設計する

将来必要だが、今回の信頼性問題を解くまでの期間が長くなる。公開・変更の安全化を先行する。

## 5. データ設計

### 5.1 `shift_publications`

シフト作成期間ごとの公開候補の版を表す。

- `id`, `shop_id`, `start_date`, `end_date`
- `state`: `draft` / `published` / `superseded` / `discarded`
- `revision`: 店舗・期間ごとに単調増加
- `created_by_staff_id`, `published_by_staff_id`
- `created_at`, `published_at`
- `source`: `ai` / `manual`

同一店舗・同一期間では公開状態を最大1版にする。公開時には現在公開版と `revision` を照合し、他の店長の変更を上書きしない。

### 5.2 `shift_draft_items`

ドラフト版に属する候補シフト。AI理由を表示文とは分離した `reason_code` で保存する。

- `publication_id`, `staff_id`, `start_datetime`, `end_datetime`, `break_time_minutes`
- `reason_code`: `requested_time`, `requested_flexible`, `fixed_candidate`, `coverage`, `manual`
- `reason_detail`（店長向け。スタッフ画面にはそのまま出さない）
- `source_request_id`（元の希望と関連付けられる場合）

### 5.3 `shift_request_decisions`

スタッフ希望ごとの判定を保存する。

- `request_shift_id`, `publication_id`, `outcome`: `accepted` / `partial` / `not_accepted` / `needs_consultation`
- `reason_code`, `staff_message`, `decided_by_staff_id`, `decided_at`

スタッフ希望は `shifts.status='requested'` と `wish_history` に残す。公開で希望行を実勤務へ変換・削除しない。休希望は `availability='rest'` として同様に保存し、必ず勤務不可制約にする。

### 5.4 `shift_change_audits`

公開済み変更の追跡専用テーブル。

- `shop_id`, `shift_id`, `publication_id`
- `actor_staff_id`, `action`: `create` / `update` / `cancel` / `publish` / `emergency_override`
- `old_value_json`, `new_value_json`, `reason_code`, `reason_note`
- `created_at`, `notified_at`, `acknowledged_at`

既存 `notifications` は表示用に使い、監査ログを通知で代替しない。

## 6. 利用者フロー

### 6.1 スタッフ

1. 募集期間内に、時間指定・柔軟希望・休希望を提出する。
2. 休希望を含む提出内容を確認し、締切前は取り消しまたは訂正する。
3. 公開後に「自分の希望 / 確定シフト / 差分理由」を見る。
4. 公開後の変更通知を受け、内容を受領する。困る場合は変更申請・相談へ進む。

### 6.2 店長

1. AI生成または手動で新しいドラフト版を作る。既存の公開シフトはこの時点で不変。
2. レビュー画面で、全体不足と個人別差分を確認する。
3. 以下のブロッカーがある場合は公開できない: 時間重複、店舗外スタッフ、月間・日次上限、休希望違反、学生の監督要件違反、必要人数超過。
4. 警告（希望外、0時間、連勤偏り、穴埋め偏り）は理由を確認して公開するか相談扱いにする。
5. 公開時、対象ドラフト版だけを一つのトランザクションで反映し、個別通知と監査ログを作る。

### 6.3 公開後の変更

- 通常変更は、旧新時刻・対象者・理由を示した差分提案として作る。
- `auto_adjust` による他者短縮は既定で禁止する。必要な場合も候補表示までとし、店長の明示承認、監査ログ、対象者通知を必要とする。
- 緊急変更は `emergency_override` と理由を必須にし、通知未受領を店長画面で追跡する。

## 7. APIと検証境界

### 新規または置換するAPI

- `POST /api/shop/publications/drafts`: 指定期間のドラフト版を作成する。生成結果を `shift_draft_items` にのみ保存する。
- `GET /api/shop/publications/<id>/review`: ブロッカー・警告・個人別差分を返す。
- `POST /api/shop/publications/<id>/publish`: 版番号を受け取り、検証・監査・通知を一つの処理で行う。
- `GET /api/staff/publications/current`: 本人の希望・確定結果・差分理由・通知受領状態を返す。
- `POST /api/staff/notifications/<id>/acknowledge`: 変更通知の受領を記録する。

既存の `/api/shop/shifts/auto` と `/api/shop/shifts/finalize` は段階的に置換する。互換期間中も `finalize` はAIドラフト以外を確定対象にしてはならない。

### 共通バリデータ

次の検証を `validate_shift_write_set(shop_id, proposed_items, existing_items)` に集約し、公開、手動追加・更新、コピー、変更申請承認のすべてから呼ぶ。

- 認証、店舗所属、退職・無効店舗
- 日時の妥当性、本人内重複、必要人数上限
- 休希望、月間・日次上限、休憩、連勤
- 学生が勤務する全スロットの監督要件

ブロッカーは書込みを止める。警告は公開者の明示確認と理由を必要とする。

## 8. セキュリティとプライバシーの前提条件

- 固定シフトの作成・更新・削除はすべて店舗管理者認証と自店舗確認を必須にする。
- 本番で `/api/init` と既知の初期パスワードを使わない。運用者向けワンタイムセットアップに置換する。
- スタッフへ返す同僚シフトから自由記述の `reason` を除外する。
- ICS用にはログインセッションではなく、失効可能・閲覧専用・期限付きの購読トークンを使う。
- AIへ渡す情報は最小化し、氏名、自由記述、個別賃金は原則送らない。送信目的とオプトアウトを明示する。

## 9. 移行

1. マイグレーションで4テーブルを追加する。既存の確定シフトと希望は変更しない。
2. リリース時点の各店舗・対象期間について、現行確定シフトのスナップショットを公開基準として作る。過去データの完全な公開履歴は作らない。
3. 新画面を有効化した店舗だけ新フローへ移す。旧 `auto/finalize` は互換期間後に廃止する。
4. 監査ログは導入後の操作から必ず取得する。

## 10. テスト戦略と受入基準

### API・単体テスト

- 未認証および他店舗から固定シフトを操作できない。
- 休希望のみの提出と、勤務希望・休希望混在の提出が保存される。
- ドラフト作成は公開済み `shifts` を変更しない。
- 公開は対象 `publication_id` の候補だけを反映し、無関係な `requested` を確定しない。
- すべての書込み経路が共通バリデータを通り、学生監督・上限・休希望を迂回できない。
- 版番号が古い公開、または検証後にデータが変わった公開は競合として拒否する。
- 公開後変更で監査ログ、通知、受領状態が作られる。

### E2Eテスト

- 「休希望を含めて提出 → AIドラフト → 差分確認 → 公開 → スタッフが理由と受領を確認」。
- 「公開後の変更 → 本人通知 → 受領 → 変更申請」。
- 「不足を埋める操作が、他スタッフの確定シフトを無通知で短縮しない」。
- モバイル幅で、取消・公開・緊急変更を誤操作しにくい導線になっている。

## 11. 第2段階への接続

この設計で確定予定と変更履歴を安定させた後、勤怠実績、月次ロック、給与計算、外部通知到達保証を別プロジェクトとして追加する。
